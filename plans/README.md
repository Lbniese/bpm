## Dependency notes

- **005 requires 002.** The dual-resolver unification is gated on the parity
  corpus (transitive/peer/override/optional/cycle/workspace/disjunctive, blocking
  vs `BPM_ASYNC_RESOLVE=1` and vs `+streaming`). Without that corpus, unifying
  the two placement cores cannot prove byte-identical output. Do not start
  005's Phase 2 until 002 has landed and is green.
- **005 benefits from 004** (registry split makes the shared registry layer
  cleaner before the async/blocking adapters converge) but does not strictly
  require it; the unification touches `registry.rs` only to add a
  `PackumentSource` abstraction, which 004 does not preclude.

  the resolver/attachment plans. They can run first and cheaply.
- **006** is independent. It touches `src/materializer.rs`, `src/volume.rs`,

  do not touch — no merge conflict expected, but run sequentially if both land
