The package manager that grows with you, not your disk.

BPM caches whole dependency graphs in a content-addressed store. Resolve the same graph twice, and the second install reuses every byte of the first — no re-download, no re-extract, no re-resolve.

The core idea

Most managers cache packages. BPM caches graphs.

Two projects that resolve to the same dependency graph share every artifact — the tarball, the extracted files, and the resolution itself. The second project's install has almost nothing left to do. Hover a node below to see it.

project-a/package.json
project-b/package.json
CONTENT-ADDRESSED STORE

Shared nodes are fetched, extracted, and verified once — every later project that resolves to them just points at the existing bytes.

Why BPM

Built on a few strict guarantees

Every one of these is a design constraint, not a nice-to-have — the store is only fast because the pieces underneath are immutable and provably correct.

Instant repeat installs

Whole dependency graphs are cached, not just individual tarballs. A second project with the same graph reuses every byte of the first.

Immutable by design

Downloaded archives, extracted files, and resolved graphs are never mutated in place — only built, verified, and published atomically.

Concurrent by default

Per-artifact locking replaces one global install lock, so multiple installs — even across projects — run safely in parallel.

Deterministic output

Lockfiles and benchmark metrics are byte-for-byte reproducible, independent of hash-map ordering, thread scheduling, or network timing.

Speaks npm's lockfile

bpm import converts an existing package-lock.json v3 straight into bpm.lock — no rewrite required.

Measured, not assumed

Every phase is instrumented. bpm bench compares against npm and pnpm with median, p95, and standard deviation.

Usage

A small, honest command set

BPM doesn't try to do everything yet — these are the commands that exist today.

bpm doctor

Inspects the nearest package.json and reports diagnostics before you install anything.

bpm fetch lodash

Resolves a package by name, version, or range (lodash, lodash@4.17.21) like npm/bun — or by exact URL — then verifies and caches it in the content-addressed store.

bpm import

Converts an existing npm package-lock.json v3 into bpm.lock.

bpm bench

Runs performance benchmark scenarios against npm and pnpm, and emits full timing results.
